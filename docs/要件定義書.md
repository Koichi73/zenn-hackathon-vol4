# 動画データから手順書の自動生成

## 課題
1. 手順書作成が手間かつ退屈な作業
2. 作成した手順書の更新が行われていない

手順書作成は、手間のかかる作業の一つである。ITリテラシーが低い人を考慮すると、全てのステップを記載したり、画面キャプチャを追加したりなど、さらに作成コストは増える。

新しいサービスの会員登録を例とすると、以下のような作業が発生する：

- 全ての画面遷移をキャプチャ
- キャプチャ内の個人情報はマスク(画像編集)
- 画像内のわかりづらいボタンは、赤枠や赤文字などで補足(画像編集)
- 全てのステップで、テキストによる手順の説明
- エラーが起きた場合のトラブルシューティングの記載
- 分岐がある場合の解説
- 最終的に手順書通りに機能するか再確認

さらに、SaaSなどアップデートが頻繁なサービスでは画面の変更が激しく、作成した手順書と異なることが発生する。しかし、手順書の更新はユーザーが手動で検知・作業する必要があり、依然として課題となっている。


## 解決策
1. 手順書作成が手間かつ退屈な作業 → 動画データからAIによる自動生成
    - ユーザーは一連の流れを画面収録する
    - AIは動画を解析して、手順書を生成する
    - 個人情報のマスクや画像内のボタンの強調なども自動で行う
    - 動画キャプチャのため、ブラウザではなくレガシーなアプリケーションでも可能
2. 作成した手順書の更新が行われていない → 利用者からの通知機能によりタイミングを把握
    - マニュアルの利用者は、実際の画面と異なる部分を簡単に報告可能
    - 作成者は、報告を受けた後、手順書を更新する
    - TODO: 利用者の挙動などからAIで自動で報告できないか(つまり、ここにもAIを使いたい)


## 競合調査 
### DOMイベントベース（Scribe, Tango）
ブラウザ拡張機能を通じてWebブラウザの裏側にあるHTMLコード（DOM）を解析し、操作ログとスクリーンショットを記録する手法
- **特徴:** テキスト生成が正確で、編集も容易
- **課題:** 
    - Figma等のCanvasアプリやリモートデスクトップ環境ではDOMが取得できず機能しない
    - プライバシー保護（マスキング）は正規表現や手動指定に依存しているため、漏洩リスクや手間の課題が残る
    - マニュアルの更新はユーザーが手動で検知・作業する必要がある

### 動画解析ベース（Guidde, Trupeer）
画面のピクセル情報（動画）をAIとOCRで解析する手法
- **特徴:** 
    - アプリケーションの種類を問わず、画面に映るものであれば何でも手順化可能
    - 出力は音声・テキスト付きの動画
- **課題:** 
    - 出力は「動画マニュアル」が主体 → 手軽な編集や閲覧が難しい

### 新興プレイヤー(Glitter AI, Clueso, Teachme Biz)
- **特徴:** 
    - 動画を構造化テキストSOPに変換する

### まとめ
競合サービスを以下の4象限マトリクスに分けることができる
- input: DOMイベントベース or 動画解析ベース
- output: テキスト or 動画


## ターゲット
「ひとり情シス vs 変化に弱い現場」

### 手順書作成者: 孤軍奮闘する「ひとり情シス」
- 佐藤さん（32歳・入社5年目）
- 従業員300名の中堅専門商社の社内SE（システム担当は彼含め2名のみ）
- マニュアル作成が退屈
- 会社が「DXだ！」と言い出し、新しいSaaS（経費精算や勤怠管理）と、自社製の古い在庫管理システム（レガシー）が混在している
- SaaSのUIが勝手にアップデートされるたびに、「画面が違うんですけど！」という内線電話が鳴り止まない
- 「スクショ撮って、赤枠つけて、パワポに貼って…この作業、今月で3回目だぞ。俺はGoogle Cloudの設計がしたいのに、なんで案内ばかりしてるんだ…」

### 手順書利用者: 変化が怖い「バックオフィス担当」
- 鈴木さん（30代半ば〜後半）
- 営業事務・経理担当。業務知識は豊富だが、PCスキルは決まった操作しかできない
- 月末は請求書処理で目が回るほど忙しい
- マニュアルは印刷して手元に置いている（更新されていない）
- マニュアル通りにクリックしようとしたらボタンがない


## コア機能
- 作成者
    - 動画をアップロードし、手順書を生成する
    - 手順書を手動で修正する
    - 手順書をアプリ上で共有する
    - 手順書をPDFでダウンロードする
    - 動画データは学習されない
- 利用者
    - 手順書をアプリ上で閲覧する
    - 手順書の古い箇所を指摘する
